# 4DreamTeam

Языки: [English](README.md) | [Русский](README.ru.md)

4DreamTeam - это волшебная палочка для тех, кто хочет работать с реальными проектами с AI-суперсилами, но не хочет превращать каждый проект в один бесконечный хрупкий чат.

Скажите Codex, что вы хотите построить, исправить, проверить, задокументировать или выпустить. 4DreamTeam превращает этот запрос в маленькую рабочую команду: одна часть уточняет цель, другая планирует работу, третья меняет код, четвертая проверяет результат, пятая поддерживает проектные заметки, а шестая помогает безопасно довести изменения до выпуска.

Вы приносите идею, хаос, недоделанную фичу или ощущение "а это вообще правда готово?". 4DreamTeam дает работе маршрут, записывает что произошло, поддерживает документацию живой и спрашивает перед опасными действиями.

## Когда Это Полезно

- У вас есть идея, но нет ТЗ.
- Вы открыли старый проект и забыли, что дальше.
- Вы хотите, чтобы Codex изменил код, но еще хотите отдельную проверку, действительно ли работа готова.
- Документация устарела, и ей уже никто не доверяет.
- Вы близко к релизу и хотите чистый план до любых действий с git.
- Вы хотите AI-помощь, которая ощущается не как один перегруженный чат, а как аккуратная проектная команда.

## Простое Обещание

```txt
идея -> план -> работа -> проверка -> документы -> релиз
```

4DreamTeam не заставляет запоминать процесс. Вы начинаете с:

```txt
$4DreamTeam
```

4DreamTeam сам направляет работу и держит следующий шаг видимым.

## Почему Это Ощущается Иначе

### У Codex появляется проектная память

Работа живет не только в прокрутке чата. 4DreamTeam записывает, что решили, что изменили, что проверили и что важно помнить, чтобы проект можно было продолжить позже.

### Он учится у оператора

Когда вы обращаете внимание на что-то важное, 4DreamTeam может превратить это в урок на будущее. Если вы объяснили, как в проекте работают тесты, какие логи только мешают, или какую ошибку не нужно повторять, агент может запомнить это и вспомнить перед похожей задачей.

### Выполнение отделено от проверки

Ассистент не просто что-то меняет и сам объявляет победу. 4DreamTeam делает отдельную проверку по исходной цели, поэтому "готово" опирается на факты, а не на ощущение на глаз.

### Документация остается рядом с реальностью

Проектные заметки остаются рядом с реальной работой. 4DreamTeam может обновлять подробную документацию из знаний о проекте, поэтому README остается простым, а глубокие материалы - полезными.

### Управление остается у вас

4DreamTeam спрашивает перед рискованными изменениями, коммитами, релизами, работой с сервером или публикацией. Можно двигаться быстро, не отдавая руль.

## Реальные Ситуации

### "Есть идея, но нет ТЗ."

```txt
Я хочу систему записи для маленькой йога-студии.
Сейчас все ведется в WhatsApp, и изменения теряются.
```

4DreamTeam помогает превратить идею в понятную цель, первый план и простой способ проверить, что работа действительно закончена, до того как Codex прыгнет в код.

### "Я вернулся через неделю."

```txt
Продолжи проект. Что дальше?
```

4DreamTeam смотрит сохраненное состояние проекта и говорит следующий безопасный шаг.

### "Нужно понять, правда ли это готово."

```txt
Фича реализована, но я не уверен, что она действительно готова.
```

4DreamTeam проверяет работу по исходной цели и записывает, прошла она проверку или нужен еще один круг.

### "Документация, наверное, устарела."

```txt
Приложение изменилось, а документация, скорее всего, уже неправильная.
```

4DreamTeam может сравнить документацию с реальным проектом, обновить то, что изменилось, и не выдумывать факты, которые он не проверял.

### "Нужно выпустить без хаоса."

```txt
Подготовь принятую работу к релизу.
```

4DreamTeam смотрит, что изменилось, обновляет описание релиза когда нужно и предлагает точный план до любых действий с git.

## Что Вы Получаете

- Помощь в превращении сырой идеи в понятную работу.
- План до начала изменений в коде.
- Изменения с понятными заметками о том, что произошло.
- Отдельную проверку перед тем, как назвать работу готовой.
- Проектные заметки, связанные с реальным исходным кодом.
- Память, которой управляет оператор: уроки, предпочтения, привычки проекта и ошибки, которые не нужно повторять.
- Документацию, которую можно выгрузить в `docs/`.
- Помощь с README, текстами для запуска, описанием релиза и проверкой обещаний.
- Серверные заметки и инструкции с проверками безопасности.
- План релиза и список изменений после вашего подтверждения.
- Способ улучшать сам 4DreamTeam через тот же аккуратный процесс.

## Быстрый Старт

Установите пакет скила `4dreamteam/` из [Tsmar/4DreamTeam](https://github.com/Tsmar/4DreamTeam/tree/main/4dreamteam).

Перезапустите Codex, затем спросите:

```txt
Is the 4DreamTeam skill available?
```

Из любой папки проекта:

```txt
Run $4DreamTeam.
```

Если папка еще не готова для 4DreamTeam, он спросит перед созданием проектных файлов.

### Работа С Существующим Проектом

Сложная настройка не нужна. Создайте чистую рабочую папку, положите репозиторий проекта внутрь `sources/`, затем запустите `$4DreamTeam`.

```txt
my-4dt-workspace/
  sources/
    your-project/
```

После настройки 4DreamTeam может изучить одобренную папку проекта, и вы готовы работать: попросите его понять проект, спланировать изменения, проверить работу, обновить документацию или подготовить релиз.

## Полезные Запросы

```txt
$4DreamTeam status
$4DreamTeam continue
$4DreamTeam validate workspace
$4DreamTeam turn this idea into a plan
$4DreamTeam check whether this feature is really done
$4DreamTeam update the project docs from source
$4DreamTeam prepare this accepted work for release
```

Проверка статуса безопасна по умолчанию. Коммиты, отправка в GitHub, теги, релизы, изменения на сервере, изменения базы данных и разрушительные действия требуют вашего явного подтверждения.

## Безопасность Простыми Словами

4DreamTeam устроен так, чтобы быть полезным без лишней безрассудности:

- он не пропускает отдельную проверку перед тем, как назвать работу готовой;
- он читает только те папки проекта, которые вы одобрили;
- он не раскрывает `.env` files, secrets, credentials или private keys;
- он спрашивает перед рискованными изменениями файлов, работой с сервером, релизами и публикацией;
- он не выдумывает обещания о продвижении, серверах или безопасности без проверенных источников.
