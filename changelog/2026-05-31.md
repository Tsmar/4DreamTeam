# 2026-05-31

Source changelog entries for this calendar date.

## 0.5.7

- Moved managed workspace state toward one SQLite authority at `.4dt/db.sqlite3`.
- Moved managed wiki authority into the shared SQLite database, with page metadata stored as indexed columns, section content, tags, and FTS rows stored in separate tables plus Markdown import for workspace conversion.
- Moved managed board authority into the shared SQLite database, with board items and timeline comments stored in separate tables plus Markdown task import for workspace conversion.
- Moved 4dt-memory default storage into the same shared `.4dt/db.sqlite3` database without importing old `.4dt/memory/db/state.sqlite3` stores, and kept memory tables single-workspace without `workspace_id` namespacing.
- Moved rebuildable 4dt-search chunks and manifests into shared `.4dt/db.sqlite3` tables and stopped writing `.4dt/search/chunks.jsonl`.
- Moved the rebuildable `4dt-wiki index` summary into the shared `.4dt/db.sqlite3` `wiki_index` table and stopped writing `.4dt/wiki/index.json`.
- Moved 4dt-sources registry, inventory, and index summaries into shared `.4dt/db.sqlite3` tables and stopped writing `.4dt/sources/registry.md` and `.4dt/sources/index/*.json`.
- Moved the rebuildable `4dt-board` index summary into shared `.4dt/db.sqlite3` table `board_index` and stopped writing `.4dt/board/.index.json`.
- Added `4dt-db backup create/list` for consistent shared database backups.
- Added full JSON export/import cycles for managed wiki, board, and memory state; imports are dry-run by default and require `--apply` to write.
- Stopped workspace bootstrap guidance from creating `sources/.gitignore`, so generated ignore rules no longer block source study through `4dt-sources`.
- Routed board creation, movement, metadata updates, and timeline comments through SQLite transactions, preserving concurrent timeline writes without racing over Markdown files.
- Kept `4dt-wiki` CLI commands stable while routing page create/update/section/apply writes through SQLite transactions and adding managed page tag commands.
- Added tag-aware page creation and `page apply` support so wiki agents can keep semantic tags aligned during imports and content updates.
- Added a rebuildable SQLite FTS5 index for wiki page titles, tags, and section content, and let `4dt-search --domain wiki` use it as the wiki candidate source before applying the shared ranking/output contract.
- Changed wiki export to render managed SQLite pages as Markdown under the requested `sources/` target.
- Added regression coverage for concurrent section writes, legacy Markdown import, section-table storage, and tag-backed search.
- Bumped the skill metadata and documented README version to `0.5.7`.

## 0.5.6

- Added a 32,000 UTF-8 byte limit for each managed wiki section and regression coverage for `4dt-wiki page section-set` and `page apply`.
- Documented that agents must serialize same-page `4dt-wiki` writes and split oversized wiki material into related managed pages.
- Stopped `4dt-wiki page section-set` and `4dt-memory keys set` write responses from echoing the saved section or contract value; explicit read commands remain available for read-back.
- Converted bundled wiki templates to the current managed frontmatter plus stable section contract.
- Removed legacy `sources.md` wiki template, the legacy `wiki_index.py` source-map script, and the `npm run wiki` alias.
- Bumped the skill metadata and documented README version to `0.5.6`.
